Auditoría UX/UI del MVP hotelero

Revisión crítica de mvp_app/ — 32 plantillas Jinja, 194 líneas de CSS y 7 blueprints de Flask — con hallazgos citados por archivo y línea, calibrados a un límite deliberado: esto es una interfaz académica mínima, no un producto comercial.

Metodología & alcance

Cómo se hizo esta auditoría

Se leyeron directamente app/templates/base.html, app/static/style.css íntegro, las 32 plantillas Jinja de los 7 módulos (auth, disponibilidad, reservas, estadía, caja, reportes, administración) y los 7 archivos routes.py más errors.py, huespedes.py y constants.py. Cada hallazgo de este informe cita archivo:línea verificable — no hay recomendaciones genéricas de plantilla.

Calibración deliberada: el profesor del curso pidió explícitamente una interfaz “no tan fina” — un MVP funcional para sustentación, no un producto comercial. Por eso cada recomendación de este informe es alcanzable con HTML/CSS/Jinja + JavaScript vanilla mínimo, sin frameworks nuevos, sin librerías de animación y sin dependencias adicionales al stack ya elegido (Flask + Jinja2). Donde una práctica de diseño 2026 exigiría más presupuesto del que el alcance académico justifica, se marca explícitamente como fuera de alcance.

Dimensión 1 / 14

Jerarquía visual y organización del contenido

1 alta 1 media

Nombres de procedimientos y vistas SQL filtran a la interfaz de usuario final

Alta
estadia/checkin_reserva.html:21disponibilidad/buscar.html:33reservas/listado.html:4caja/ver_cuenta.html:12
Qué está mal

~16 de 27 botones y ~10 de los ~20 encabezados incluyen literalmente el nombre del SP/vista entre paréntesis: "Realizar check-in (sp_realizar_checkin)", <h1>Reservas (vw_reservas_detalle)</h1>.

Por qué afecta al usuario

Un recepcionista o cajero no necesita ni entiende qué es un "sp_" o una "vw_" — es jerga de implementación filtrada a la capa de presentación. Además es inconsistente: ni siquiera aparece en todos los botones equivalentes, así que tampoco funciona como ayuda de depuración sistemática.

Cómo solucionarlo

Mover esa referencia a un comentario Jinja {# invoca sp_realizar_checkin #} (invisible en el HTML renderizado) o a un atributo data-sp="sp_realizar_checkin" solo visible para quien inspeccione el DOM durante la sustentación.

Antes · checkin_reserva.html:21
<button type="submit">
  Realizar check-in
  (sp_realizar_checkin)
</button>
Después
{# invoca sp_realizar_checkin #}
<button type="submit">
  Realizar check-in
</button>

Tono inconsistente en títulos H1/H2

Media
reservas/detalle.html:2administracion/hoteles.html:4disponibilidad/buscar.html:4
Qué está mal

El <h1> de cada pantalla mezcla estilos sin convención: a veces es el nombre de la entidad ("Hoteles"), a veces una frase de acción ("Consultar disponibilidad"), a veces datos interpolados ("Reserva #12 — Hotel San Marcos Lima").

Por qué afecta al usuario

La jerarquía visual (CSS) es consistente, pero la convención de contenido no lo es — el usuario no puede predecir qué tipo de información encontrará en el título de la siguiente pantalla.

Cómo solucionarlo

Adoptar una convención fija: [Entidad] — [dato clave] para pantallas de detalle, y solo [Entidad en plural] para listados. Ya existe .page-header en el CSS (usado hoy solo en el dashboard) — extenderlo a las 19 pantallas restantes resuelve esto y el hallazgo de navegación #2 a la vez.

Dimensión 2 / 14

Distribución de los elementos (layout y espaciado)

2 media

Una sola clase .card para formularios y paneles de solo lectura

Media
style.css:96-104reservas/detalle.html:6-11caja/ver_cuenta.html:5-9
Qué está mal

.card se reutiliza tanto para form.card (algo que el usuario debe completar) como para <div class="card"> de datos ya guardados (algo que el usuario solo lee) — misma apariencia, dos propósitos distintos.

Por qué afecta al usuario

El usuario debe leer el contenido para saber si una tarjeta es "editable" o "informativa" — no hay ninguna señal visual (borde, ícono, encabezado de sección) que lo distinga de un vistazo.

Cómo solucionarlo

Agregar un modificador .card--readonly con fondo ligeramente distinto (var(--surface-2) en vez de blanco puro) para los paneles de solo lectura, sin tocar la clase base.

Páginas con 3–5 formularios apilados sin jerarquía entre ellos

Media
estadia/ver.html:29-118reservas/nuevo.html:6-107
Qué está mal

La pantalla "hub" de un alojamiento (estadia/ver.html) apila 5 form.card completos en una sola columna larga: agregar huésped, crear huésped, registrar consumo, registrar daño, checkout. reservas/nuevo.html hace lo mismo con 2 formularios completos (cabecera de reserva + alta de cliente).

Por qué afecta al usuario

Todo tiene el mismo peso visual, así que la acción más frecuente (agregar un consumo) no se distingue de la menos frecuente (dar de alta un huésped completamente nuevo, un caso excepcional). El usuario debe leer toda la página para orientarse.

Cómo solucionarlo

Usar el elemento nativo <details><summary> (cero JS, cero dependencias) para colapsar por defecto los formularios secundarios/excepcionales ("¿Huésped nuevo?"), dejando expandido solo el flujo principal.

Antes
<h2>¿Huésped nuevo?</h2>
<form class="card">...</form>
Después
<details class="card">
  <summary>¿Huésped nuevo?</summary>
  <form>...</form>
</details>
Dimensión 3 / 14

Paleta de colores, contraste y legibilidad

1 fortaleza 1 media

Fortaleza confirmada: el contraste de color ya cumple WCAG AA

Fortaleza
style.css:1-16
Qué está bien

Se calcularon los ratios reales de las combinaciones de la paleta: texto muted sobre fondo (5.43:1), texto muted sobre tarjeta (5.98:1), texto base sobre fondo (13.65:1), blanco sobre navy (12.67:1), alerta de éxito (7.03:1) y de error (8.43:1) — todas pasan AA, la mayoría con margen para AAA.

Por qué es relevante

Esto no es un área a priorizar en el rediseño — el esfuerzo debe ir a accesibilidad estructural (foco, semántica, formularios), no a repintar colores.

Estados de negocio sin codificación de color (texto plano)

Media
style.css:149 (.badge sin uso)estadia/activos.htmlcaja/cuentas.html
Qué está mal

Estados como DISPONIBLE/OCUPADA/RESERVADA/LIMPIEZA o PENDIENTE/PAGADA se muestran como texto negro plano en las tablas. La clase .badge ya existe en el CSS pero no se usa en ninguna plantilla.

Por qué afecta al usuario

En una tabla de 20 habitaciones, distinguir cuáles están OCUPADAS de cuáles están DISPONIBLES exige leer palabra por palabra en vez de escanear por color — el patrón de "estado como color" (chip/pill) que pide Material Design 3 no está aplicado pese a que el CSS ya lo soporta.

Cómo solucionarlo

Mapear cada estado a una clase de severidad ya definida (.badge.alta/media/baja/good) vía un pequeño filtro Jinja, sin CSS nuevo.

Antes · estadia/activos.html
<td>{{ h.estado }}</td>
Después
<td>
  <span class="badge {{ estado_color[h.estado] }}">
    {{ h.estado }}
  </span>
</td>

Qué está bien

Todos los tamaños de fuente usan rem (nunca px), lo que respeta el zoom del usuario — una buena práctica de accesibilidad frecuentemente ignorada. line-height: 1.45 global da buen espacio de lectura.

Sin text-wrap: balance en encabezados

Baja
style.css:56-58
Qué está mal

Encabezados largos con interpolación de datos (ej. "Pre-asignación de huéspedes — Reserva #12") pueden partirse en líneas desbalanceadas.

Cómo solucionarlo

Dimensión 5 / 14

Navegación y facilidad para encontrar información

2 alta 1 media

Sin indicador de "sección activa" en el menú

Alta
base.html:12-33
Qué está mal

Ningún enlace del <nav> tiene aria-current="page" ni una clase .active — se confirmó por búsqueda global, cero coincidencias.

Por qué afecta al usuario

El rol Administrador ve 7 enlaces con apariencia idéntica; nada indica en cuál de ellos está parado. Es la carencia de orientación más notoria de toda la app.

Cómo solucionarlo

Comparar request.endpoint contra cada enlace en Jinja y añadir aria-current="page" + una clase visual.

Antes · base.html
<a href="{{ url_for(
  'administracion.hoteles') }}">
  Hoteles</a>
Después
<a href="{{ url_for(
  'administracion.hoteles') }}"
  {% if request.endpoint ==
     'administracion.hoteles' %}
   class="active" aria-current="page"
  {% endif %}>Hoteles</a>

Flujo de 4 pasos (reserva) sin ruta ni indicador de progreso

Alta
style.css:60-68 (.breadcrumb, sin uso)reservas/nuevo.html:4
Qué está mal

El CSS ya define .page-header .breadcrumb, pero ninguna plantilla la usa. El flujo cabecera → detalle → pago → pre-asignación no muestra en qué paso está el usuario ni cuántos quedan; el <h1> solo dice "paso 1" sin indicar el total.

Por qué afecta al usuario

En un flujo transaccional de varias pantallas, no saber "cuánto falta" genera ansiedad e incertidumbre — un principio básico de UX de formularios largos.

Cómo solucionarlo

Activar el breadcrumb ya definido en CSS: Reserva #12 › Detalle › Pago, con el paso actual en negrita — cero CSS nuevo, solo conectarlo en las 4 plantillas.

Tres patrones distintos de filtrado para el mismo propósito

Media
reservas/listado.html:5-10estadia/historial.html:5-9reportes/ocupacion.html:16-24
Qué está mal

Filtrar un listado se resuelve de 3 formas distintas en la misma app: enlaces con querystring fija, un formulario GET con botón "Filtrar", y un <select onchange="window.location=..."> de auto-navegación.

Por qué afecta al usuario

El tercer patrón (auto-submit al cambiar el select) es además un antipatrón de accesibilidad documentado (WCAG 3.2.2, "On Input"): un usuario que navega el select con las flechas del teclado dispara una navegación no solicitada, sin previo aviso.

Cómo solucionarlo

Adoptar un único patrón (formulario GET + botón explícito "Filtrar") en las 3 pantallas.

Dimensión 6 / 14

Diseño responsive (móvil, tablet, escritorio)

2 alta

Cero meta viewport y cero media queries en todo el proyecto

Alta
base.html:1-7style.css (194 líneas, 0 @media)
Qué está mal

El <head> no declara <meta name="viewport">, y el CSS no tiene ni una sola @media query. Sin viewport, un móvil renderiza a ~980px de ancho virtual y fuerza al usuario a hacer zoom/pan.

Por qué afecta al usuario

Un recepcionista revisando una reserva desde su celular (escenario realista en un hotel) tiene una experiencia de escritorio comprimida, no una app usable.

Cómo solucionarlo

Agregar la meta tag (una línea) y 2–3 media queries básicas para colapsar el grid de KPIs y el nav en pantallas angostas.

Antes · base.html <head>
<meta charset="utf-8">
<title>...</title>
Después
<meta charset="utf-8">
<meta name="viewport"
  content="width=device-width,
           initial-scale=1">
<title>...</title>

Tablas sin contenedor de scroll horizontal

Alta
style.css:70-94~20 plantillas con <table>
Qué está mal

Ninguna de las ~20 tablas (ej. vw_reservas_detalle con 10 columnas) está envuelta en un contenedor con overflow-x: auto.

Por qué afecta al usuario

En pantalla angosta, una tabla de 10 columnas se comprime ilegible o desborda la página sin forma controlada de desplazarse horizontalmente.

Cómo solucionarlo

Envolver cada <table> en <div class="table-scroll"> con overflow-x:auto — un solo patrón, reutilizable en las 20 plantillas.

Dimensión 7 / 14

Accesibilidad (WCAG): contraste, foco, teclado, etiquetas

4 alta

label for= sin id correspondiente en ~90% de los formularios

Alta
reservas/nuevo.html:7-8,54-55disponibilidad/buscar.html:7-8estadia/ver.html:42-76administracion/*.html
Qué está mal

Los <label for="X"> apuntan a un id="X" que casi nunca existe en el control (solo tiene name="X"). Único caso correcto: auth/login.html. Caso incluso peor en reservas/nuevo.html:54-55: el label for="tipo" ni siquiera coincide con el id="tipo-cliente" real del select.

Por qué afecta al usuario

Un lector de pantalla no anuncia la etiqueta al enfocar el campo, y clicar sobre el texto de la etiqueta no enfoca el input — falla dos veces la accesibilidad de formularios (WCAG 1.3.1 / 4.1.2).

Cómo solucionarlo

Agregar id igual al name en cada control — sin cambiar CSS ni lógica de backend.

Antes · disponibilidad/buscar.html:7-8
<label for="id_hotel">Hotel</label>
<select name="id_hotel" required>
Después
<label for="id_hotel">Hotel</label>
<select name="id_hotel" id="id_hotel" required>

Ninguna tabla usa <thead>/<tbody>

Alta
~20 plantillas, 0 coincidencias de <thead>
Qué está mal

Todas las tablas usan una <tr><th>...</th></tr> plana seguida del bucle Jinja, sin separación semántica de encabezado/cuerpo.

Por qué afecta al usuario

Los lectores de pantalla usan <thead> para anunciar "encabezado de columna X" al navegar celda por celda — sin él, la tabla se lee como texto plano sin contexto de columna.

Cómo solucionarlo

Envolver la fila de encabezado en <thead> y el {% for %} en <tbody> — cambio mecánico, mismo CSS.

Foco de teclado visible solo en inputs, no en enlaces/botones

Alta
style.css:117-121
Qué está mal

Solo input, select, textarea definen un anillo de foco (borde + sombra); ningún <a> ni <button> tiene :focus propio, solo :hover.

Por qué afecta al usuario

Un usuario que navega solo con Tab por el menú de 7 enlaces (rol Administrador) no tiene una señal de foco diseñada — depende del outline nativo del navegador, que puede casi no distinguirse contra el degradado oscuro del topbar.

Cómo solucionarlo

Una regla global con :focus-visible (ya usada en este mismo informe) que no interfiera con el mouse pero sí sea nítida con teclado.

Agregar a style.css
a:focus-visible,
button:focus-visible {
  outline: 2px solid var(--color-primary);
  outline-offset: 2px;
}

Flash messages sin role="alert" ni "skip to content"

Alta
partials/flash.html:3-7base.html (sin skip-link)
Qué está mal

Los mensajes de error/éxito no tienen role="alert"/aria-live, así que un lector de pantalla no los anuncia automáticamente al cargar la página. Tampoco existe un enlace "Saltar al contenido" antes del nav.

Cómo solucionarlo

<div class="flash-container" role="alert"> (una palabra) + un <a class="skip-link" href="#main">Saltar al contenido</a> visualmente oculto hasta recibir foco.

Dimensión 9 / 14

Flujo de usuario y reducción de fricción

3 alta

El hallazgo de mayor impacto: los errores de negocio vacían el formulario completo

Alta · máxima prioridad
reservas/routes.py:112-122reservas/routes.py:167-173reservas/routes.py:97disponibilidad/routes.py:31-57 (patrón correcto)
Qué está mal

Cuando un procedimiento almacenado rechaza una operación (SIGNAL — ej. sobreventa, RUC duplicado, capacidad excedida), el código casi siempre hace redirect() a la página en blanco en vez de re-renderizarla con los datos ya escritos. La única excepción correcta ya existe en disponibilidad/routes.py::buscar_post, que sí pasa seleccion=request.form de vuelta a la plantilla.

Por qué afecta al usuario

Un recepcionista que se equivoca en la cantidad de habitaciones de una reserva de 8 campos, o en el alta de un cliente jurídico de 14 campos, pierde todo lo tecleado y debe empezar de cero al ver el mensaje de error. Es la fricción más severa medida en todo el sistema.

Cómo solucionarlo

Replicar el patrón ya correcto de disponibilidad en reservas, estadia, caja y administracion: cuando ejecutar_con_flash devuelva ok=False, re-renderizar el template con request.form en vez de redirigir.

Antes · reservas/routes.py:118-122
ok, data = ejecutar_con_flash(
    call_procedure, "sp_registrar_reserva",
    (...), on_success_msg="...")
if not ok:
    return redirect(
      url_for("reservas.nuevo"))
Después · mismo patrón que disponibilidad
ok, data = ejecutar_con_flash(
    call_procedure, "sp_registrar_reserva",
    (...), on_success_msg="...")
if not ok:
    return render_template(
      "reservas/nuevo.html",
      clientes=clientes, hoteles=hoteles,
      canales=CANALES, seleccion=request.form)

Login forzado no conserva la página de destino

Alta
auth/routes.py:16-20
Qué está mal

requiere_rol siempre redirige a /login sin un parámetro ?next=; tras loguearse, el usuario cae en el dashboard, no en la página que originalmente pedía.

Por qué afecta al usuario

Un enlace profundo compartido entre colegas (ej. a una reserva específica) se pierde si el receptor no tenía sesión activa — debe volver a navegar manualmente hasta ahí.

Cómo solucionarlo

Guardar request.path en session['next'] antes de redirigir, y usarlo en login() tras autenticar.

Confirmación de acciones irreversibles: solo 1 de al menos 5 casos sensibles

Alta
estadia/ver.html:12 (único confirm())reservas/pago.html:11-13caja/ver_cuenta.html (pago)
Qué está mal

Solo el checkout grupal pide confirmación (onsubmit="return confirm(...)"). Confirmar pago, registrar pago de cuenta, registrar salida individual de huésped y cambiar el estado de una habitación — igual de sensibles o más — no piden ninguna confirmación.

Por qué afecta al usuario

No hay un criterio aparente de cuándo el sistema protege al usuario de un clic accidental; se siente arbitrario en vez de una política de diseño.

Cómo solucionarlo

Aplicar el mismo confirm() (o, mejor, un <dialog> nativo, ver Dimensión 14) de forma consistente a las 4 acciones restantes.

Dimensión 11 / 14

Estados de la interfaz: hover, focus, loading, vacío, error, éxito

3 alta

13 de 14 listados no manejan el estado vacío

Alta
caja/generar_cuenta.html:15-17 (único caso correcto)reservas/listado.html, estadia/*, reportes/*, administracion/* (sin manejo)
Qué está mal

Si la colección de una tabla está vacía, la tabla se renderiza solo con la fila de encabezado y el cuerpo en blanco — sin mensaje de "no hay datos". El único caso bien resuelto es caja/generar_cuenta.html.

Por qué afecta al usuario

Con los datos de prueba limitados de un MVP académico, es muy probable que un evaluador tope con un listado vacío durante la sustentación y lo interprete como un error, no como "no hay resultados aún".

Cómo solucionarlo

Replicar el patrón ya correcto: {% if not filas %}<p>No hay [entidad] registrada(s).</p>{% endif %} en las 13 plantillas restantes.

Errores de negocio: solo banner genérico, nunca junto al campo

Alta
app/errors.py:14-21partials/flash.html:3-7
Qué está mal

ejecutar_con_flash muestra el MESSAGE_TEXT del SIGNAL tal cual, como único banner arriba de la página — nunca junto al campo del formulario que causó el problema (no existe ningún patrón de error a nivel de campo en toda la app).

Por qué afecta al usuario

Cómo solucionarlo

Sin protección de doble-submit en pagos y checkout

Alta

Qué está mal

No hay spinner, ni texto "Procesando…", ni disabled tras el primer clic, en ninguna acción — incluidas "Confirmar pago" y "Registrar pago", las más sensibles del sistema.

Por qué afecta al usuario

Un doble clic accidental (frecuente sin feedback visual de que algo está pasando) podría intentar duplicar un pago, dependiendo de si el procedimiento lo previene a nivel de base de datos.

Cómo solucionarlo

3 líneas de JS vanilla, sin dependencias: deshabilitar el botón y cambiar su texto al enviar el formulario.

Agregar (vanilla, sin dependencias)
document.querySelectorAll('form').forEach(f =>
  f.addEventListener('submit', () => {
    const b = f.querySelector('button[type=submit]');
    if (b) { b.disabled = true; b.textContent = 'Guardando…'; }
  }));
